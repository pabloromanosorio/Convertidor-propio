Create `src/editor/consistency_agent.js`:

**Function: `runConsistencyCheck(store, modelKey, log)`**

This runs AFTER all sections have been analyzed but BEFORE the user does final approval. It uses the LLM to detect inconsistencies.

**Step 1: Group findings by similarity**
- Get all findings from store
- Group by: exact `findText` match, OR same `errorType` + similar `errorPhrase` (fuzzy match using normalized lowercase comparison — if 70%+ of words overlap)
- Each group represents a "recurring pattern"

**Step 2: Identify conflicts within groups**
For each group with 2+ findings:
- Compare `replaceText` values. If they differ, this is a POTENTIAL INCONSISTENCY.
- Not all differences are conflicts — different sentence contexts legitimately require different corrections (gender agreement, plural, etc.). The LLM will decide.

**Step 3: LLM consistency review**
For groups with differing `replaceText` values, send to LLM:

```
You are a translation consistency reviewer.

The following corrections were proposed for similar errors across different sections of the same document. Check if they are CONSISTENT or CONTRADICTORY.

A correction is CONSISTENT if:
- Different wording is justified by different grammatical context (gender, number, verb tense)
- The same underlying term is used but conjugated/declined appropriately

A correction is CONTRADICTORY if:
- The same source term is translated differently without grammatical justification
- Two fixes propose fundamentally different approaches to the same error pattern
- One fix uses formal register while another uses informal for the same type of text

FINDINGS TO CHECK:
${group.findings.map(f => `
Section ${f.sectionIndex}, Segment [${f.segmentId}]:
  Error: "${f.errorPhrase}"
  Context: "${f.contextPhrase}"
  Find: "${f.findText}"
  Replace: "${f.replaceText}"
  Source: "${f.sourcePhrase}"
`).join('\n')}

For each contradiction found, return a JSON object:
{
  "contradictions": [
    {
      "findingIds": ["id1", "id2"],
      "reason": "why these contradict each other",
      "recommendation": "which approach to use and why",
      "recommendedReplaceText": "the consistent replacement to use"
    }
  ],
  "consistent": true/false
}

If all corrections are consistent (just adapted to context), return: { "contradictions": [], "consistent": true }
```

**Step 4: Store consistency results**
- Add consistency warnings to the store: `store.addConsistencyWarnings(warnings)`
- Each warning links to the affected finding IDs
- Findings with contradictions get `consistencyWarning: { reason, recommendation, recommendedReplaceText }`

**Step 5: Return summary**
Return `{ totalGroups, contradictionsFound, warnings: [...] }`

**Skip groups with only 1 finding** — no consistency to check.
**Skip groups where all replaceText values are identical** — already consistent.
Only send groups with actual differences to the LLM (saves tokens).

**Create `src/editor/correction_applier.js`:**

**Function: `applyCorrectionsToXliff(store, log)`**

1. Get all approved corrections via `store.getApprovedCorrections()` — returns `[{ segmentId, findText, replaceText }]`
2. Sort corrections by segmentId (group corrections for the same segment together)
3. For each segment with corrections:
   a. Get current target text from `store.segments.find(s => s.id === segmentId).target`
   b. Apply all corrections for this segment IN ORDER (by position in text — find the index of findText, apply from end to start to avoid offset shifting)
   c. **CRITICAL: Before applying, verify findText still exists in the current target.** If a prior correction in the same segment changed the text, findText might no longer match. If no match:
      - Try fuzzy match (ignore whitespace differences, case)
      - If still no match, log warning: "Could not apply correction for segment {id}: '{findText}' not found in target after prior corrections"
      - Skip this correction and flag it as `applicationFailed: true`
   d. Update `store.segments[i].target` with corrected text
4. After all corrections applied in memory, call `mergeXliffTranslations(store.workingXliffPath, store.segments)` from xliff_handler.js — this writes corrected targets into the XLIFF file
5. Return `{ applied: count, failed: count, failedDetails: [...] }`

**Function: `exportCorrectedDocx(store, outputPath, log)`**

1. Ensure corrections have been applied (check `store.correctionsApplied` flag)
2. Call `convertToDocx(store.workingXliffPath, outputPath)` from xliff_handler.js
3. This uses OpenXLIFF Java to merge the XLIFF (with corrected targets) back into a DOCX using the embedded skeleton
4. The skeleton contains ALL formatting — we only changed text in `<target>` elements
5. Return `{ outputPath, success: true }`

**Apply-from-end strategy for multiple corrections in same segment:**
If a segment has corrections at positions 10-20 and 50-60, apply the one at 50-60 first. This prevents position shifting from invalidating later corrections. Implementation:
```javascript
// Find positions of each correction's findText in the target
const positions = corrections.map(c => ({
  ...c,
  index: target.indexOf(c.findText)
})).filter(c => c.index !== -1);
// Sort by position DESCENDING (apply from end to start)
positions.sort((a, b) => b.index - a.index);
// Apply in order
for (const c of positions) {
  target = target.substring(0, c.index) + c.replaceText + target.substring(c.index + c.findText.length);
}
```

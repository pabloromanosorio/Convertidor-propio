Output: `src/editor/xliff_aligner.js`, `src/editor/segment_store.js`, new route `POST /api/editor/upload`
</objective>

Create `src/editor/` directory and `xliff_aligner.js` module. This module wraps the existing `xliff_handler.js` functions for the editor use case.

**Function: `alignDocxPair(sourceDocxPath, targetDocxPath, options)`**
- Creates a temp directory via `os.tmpdir()` + unique subfolder
- Converts SOURCE DOCX to XLIFF using `convertToXliff(sourceDocxPath, tempDir, options.srcLang || 'en')` — this produces the skeleton + source segments
- Converts TARGET DOCX to XLIFF similarly using `convertToXliff(targetDocxPath, tempDir2, options.tgtLang || 'es')`
- Extracts segments from both XLIFF files using `extractXliffSegments()`
- Aligns segments by position (index-based — OpenXLIFF produces segments in document order, so segment N of source corresponds to segment N of target)
- For each aligned pair, creates: `{ id: segment.id, unitId: segment.unitId, source: sourceSegment.source, target: targetSegment.source, sectionIndex: computedSection }`
- **Section computation:** Detect section breaks by looking for segments where source text is ALL CAPS, or is short (<100 chars) and followed by longer text — these are likely headings/section titles. Group segments between headings into sections. If no headings detected, chunk every ~20 segments into a section.
- Returns `{ segments: [...], sourceXliffPath, targetXliffPath, tempDir, metadata: { totalSegments, sectionCount, srcLang, tgtLang } }`

**Function: `loadExistingXliff(xliffPath)`**
- Calls `extractXliffSegments(xliffPath)` from xliff_handler
- Returns segments with both source and target already populated (from a prior translation run)
- Applies same section detection logic
- Returns `{ segments, xliffPath, metadata }`

**Function: `getSegmentsWithContext(segments, sectionIndex)`**
- Returns all segments for a given section
- Adds `contextPrevious` (last 2 segments of prior section) and `contextNext` (first 2 segments of next section) for LLM context
- Returns `{ sectionSegments, contextPrevious, contextNext, sectionTitle }`

**IMPORTANT:** The aligner must copy the TARGET DOCX's XLIFF file as the "working XLIFF" — this is the file that will be modified with corrections and merged back to DOCX. The source XLIFF is only used for extracting source text. Store the working XLIFF path in the return object as `workingXliffPath`.

Actually, for the two-DOCX case, the cleanest approach is:
1. Convert source DOCX to XLIFF (gives us source segments + skeleton)
2. Extract target DOCX segments separately (just for text alignment)
3. Merge target text INTO the source XLIFF's `<target>` elements using `mergeXliffTranslations()`
4. Now we have one bilingual XLIFF with skeleton from source + targets from translation
5. This XLIFF is the working file — corrections go into its targets
6. Final `convertToDocx()` produces corrected DOCX from this working XLIFF

Use this approach. It guarantees the skeleton matches the source document structure.

Handle edge cases:
- Different segment counts between source and target: Log warning, align as many as possible, mark unaligned segments with `aligned: false`
- Empty segments: Skip segments with empty source AND empty target
- XLIFF inline tags: Preserve `<bpt>`, `<ept>`, `<ph>` tags in source/target text (they represent formatting)

**Create `src/editor/segment_store.js`:**

Class `SegmentStore` that holds the editor session state:

```
constructor(alignmentResult) {
  this.segments = alignmentResult.segments
  this.workingXliffPath = alignmentResult.workingXliffPath || alignmentResult.xliffPath
  this.tempDir = alignmentResult.tempDir
  this.metadata = alignmentResult.metadata
  this.findings = []         // populated by proofreading pipeline
  this.corrections = new Map() // segmentId -> { original, corrected, approved }
  this.sessionId = crypto.randomUUID()
}
```

Methods:
- `getSections()` — returns array of `{ index, title, segmentCount, startId, endId }`
- `getSectionSegments(sectionIndex)` — returns segments for that section with context
- `addFindings(sectionIndex, findings)` — stores LLM findings for a section
- `getFindingsGrouped()` — returns findings grouped by term/pattern (group findings that reference the same source term or same error pattern across sections)
- `approveCorrection(findingId)` — marks a finding as approved
- `rejectCorrection(findingId)` — marks a finding as rejected
- `getApprovedCorrections()` — returns all approved corrections as `[{ segmentId, original, replacement }]`
- `applyCorrections()` — applies approved corrections to segment targets in memory
- `toJSON()` — serializable state for frontend

**Grouping logic for `getFindingsGrouped()`:**
- Group findings where the `findText` (the erroneous phrase) matches exactly, OR where the `errorType` + normalized key words overlap >60%
- Within each group, sort by section order (document position)
- Each group has: `{ groupId, term, errorType, findings: [...] }`

**Add to `server.js`:**

Store active editor sessions in a Map: `const editorSessions = new Map()`

Add route `POST /api/editor/upload`:
- Accept multipart form with fields: `sourceFile` (DOCX), `targetFile` (DOCX), OR `xliffFile` (XLF)

- Return segments for that section with source and target text

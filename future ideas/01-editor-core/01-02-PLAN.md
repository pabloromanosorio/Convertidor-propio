**Create `src/editor/finding_schema.js`:**

Define the structured finding format:

```javascript
const SEVERITY_LEVELS = ['critical', 'major', 'medium', 'minor', 'preferential'];

const ERROR_TYPES = [
  'mistranslation',           // Wrong meaning
  'omission',                 // Missing content
  'addition',                 // Added content not in source
  'untranslated',             // Left in source language
  'terminology',              // Wrong term for domain
  'grammar',                  // Grammar error in target
  'syntax',                   // Awkward/clunky sentence structure
  'punctuation',              // Punctuation issues
  'number_format',            // Number/date format errors
  'register',                 // Wrong formality level
  'inconsistency',            // Same term translated differently
  'style',                    // Style/flow issues
  'spelling',                 // Spelling errors
  'capitalization',           // Capitalization errors
];

// Finding structure:
// {
//   id: string (UUID),
//   sectionIndex: number,
//   segmentId: string (XLIFF segment ID),
//   severity: SEVERITY_LEVELS[n],
//   errorType: ERROR_TYPES[n],
//   errorPhrase: string,          // exact erroneous phrase in target
//   contextPhrase: string,        // full sentence/phrase containing the error for context
//   sourcePhrase: string,         // corresponding source phrase for reference
//   explanation: string,          // why this is wrong and what the fix does
//   findText: string,             // exact text to find in target segment
//   replaceText: string,          // replacement text (contextually adapted)
//   approved: null | boolean,     // null=pending, true=approved, false=rejected
//   groupId: null | string,       // set later by consistency agent
//   verificationNote: null | string  // set later by web search verification
// }
```

Export `validateFinding(finding)` — checks all required fields are present and values are in the allowed enums. Returns `{ valid, errors }`.

**Create `src/editor/proofreader.js`:**

**Function: `analyzeSection(store, sectionIndex, modelKey, customPrompt, log)`**

1. Get section segments via `store.getSectionSegments(sectionIndex)` — returns `{ sectionSegments, contextPrevious, contextNext, sectionTitle }`

2. Build the analysis prompt. This is THE critical prompt. Structure it as:

```
You are a senior translation quality reviewer specializing in {srcLang}→{tgtLang} translation.

TASK: Analyze the following source-target segment pairs for translation errors. For each error found, produce a structured finding.

${customPrompt ? `ADDITIONAL REVIEWER INSTRUCTIONS:\n${customPrompt}\n` : ''}

CONTEXT FROM PREVIOUS SECTION (for continuity, do NOT analyze these):
${contextPrevious.map(s => `[${s.id}] SRC: ${s.source}\n     TGT: ${s.target}`).join('\n')}

SECTION TO ANALYZE — "${sectionTitle}":
${sectionSegments.map(s => `[${s.id}] SRC: ${s.source}\n     TGT: ${s.target}`).join('\n')}

CONTEXT FROM NEXT SECTION (for continuity, do NOT analyze these):
${contextNext.map(s => `[${s.id}] SRC: ${s.source}\n     TGT: ${s.target}`).join('\n')}

For each error found, output a JSON object with these fields:
- segmentId: the [id] of the segment containing the error
- severity: one of "critical", "major", "medium", "minor", "preferential"
- errorType: one of "mistranslation", "omission", "addition", "untranslated", "terminology", "grammar", "syntax", "punctuation", "number_format", "register", "inconsistency", "style", "spelling", "capitalization"
- errorPhrase: the EXACT problematic phrase as it appears in the target text
- contextPhrase: a longer phrase from the target (full clause or sentence) that contains the error, so a reviewer can see it in context
- sourcePhrase: the corresponding phrase from the source text
- explanation: a clear explanation of WHY this is an error and what the correction achieves. Be specific about grammar rules, terminology standards, or style guidelines.
- findText: the EXACT text to search for in the target segment (must be a verbatim substring of the target)
- replaceText: the corrected text to replace it with. IMPORTANT: The replacement must be ORGANIC — adapt gender, number, articles, prepositions, verb conjugations, and surrounding syntax as needed so the corrected sentence reads naturally. Do NOT do a mechanical word swap that would introduce new errors.

CRITICAL RULES:
1. findText MUST be an exact substring of the target text for that segment. If you cannot find an exact match, expand findText to include more surrounding words until it is unique and exact.
2. replaceText must produce a grammatically correct, natural-sounding sentence when substituted into the target.
3. If fixing one error requires changing multiple parts of the same sentence (e.g., changing a noun requires changing its article and adjective), include ALL changes in a single finding with findText/replaceText spanning the full affected phrase.
4. Do NOT report stylistic preferences as errors unless the phrasing is genuinely awkward or misleading.
5. For terminology errors, note the correct domain-specific term.
6. Severity guide:
   - critical: Changes meaning, causes legal/safety/medical risk
   - major: Significant meaning change or omission
   - medium: Grammar/syntax error that hurts readability
   - minor: Small grammatical issue, formatting
   - preferential: Style preference, equally valid alternatives

Output format: Return a JSON array of findings. If no errors found in this section, return an empty array [].
Return ONLY the JSON array, no other text.
```

3. Call `generateTextOnly(modelKey, prompt, log, { temperature: 0.1 })` from translator.js

4. Parse the LLM response:
   - Strip markdown code fences if present (```json ... ```)
   - `JSON.parse()` the response
   - Validate each finding with `validateFinding()`
   - Verify each `findText` is actually a substring of the target text in the referenced segment — if not, log a warning and mark finding as `findTextVerified: false`
   - Assign UUID to each finding
   - Set `sectionIndex`, `approved: null`, `groupId: null`

5. Store findings via `store.addFindings(sectionIndex, validatedFindings)`

6. Return `{ findings: validatedFindings, sectionIndex, errors: parsingErrors }`

**Function: `analyzeFull(store, modelKey, customPrompt, log)`**
- Iterates all sections from `store.getSections()`
- Calls `analyzeSection()` for each section sequentially (not parallel — to avoid rate limits and allow progressive UI updates)
- After each section, broadcast progress via `log()`
- Returns `{ totalFindings, sectionsAnalyzed, errors }`

**Retry logic:**
- If JSON parse fails, retry up to 3 times with a note in the prompt: "Your previous response was not valid JSON. Please return ONLY a JSON array."
- If <50% of findText matches verify, retry with: "Several of your findText values were not exact substrings of the target. Please double-check each findText against the target text."

Add these routes to `server.js`:

**`POST /api/editor/analyze`**
- Body: `{ sessionId, modelKey, customPrompt, sectionIndex? }`
- If `sectionIndex` provided: analyze single section via `analyzeSection()`
- If `sectionIndex` not provided: analyze all sections via `analyzeFull()`
- Broadcast progress via SSE: "📝 Analyzing section {n}/{total}..." for each section
- Return `{ findings: [...], totalFindings, sectionsAnalyzed }`

**`GET /api/editor/session/:sessionId/findings`**
- Returns all findings for the session, grouped by term/pattern
- Calls `store.getFindingsGrouped()`
- Returns `{ groups: [...], totalFindings, approvedCount, rejectedCount, pendingCount }`

**`GET /api/editor/session/:sessionId/section/:sectionIndex/findings`**
- Returns findings for a specific section (ungrouped, in document order)

**`POST /api/editor/session/:sessionId/finding/:findingId/approve`**
- Marks finding as approved
- Returns updated finding

**`POST /api/editor/session/:sessionId/finding/:findingId/reject`**
- Marks finding as rejected
- Returns updated finding

**`POST /api/editor/session/:sessionId/approve-group/:groupId`**
- Approves all findings in a group at once
- Returns updated group

Import the proofreader module at the top of server.js. Validate sessionId exists in editorSessions map for all /editor/ routes (return 404 if not found).
